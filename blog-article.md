## 第一阶段：项目初始化（23:13 - 23:46）

### 我的输入

> 帮我做一个类似多邻国风格的幼儿英语学习 App，面向 3-6 岁儿童，iPad 优先，支持 PWA 离线使用。

就这么一句话。没有画原型，没有写需求文档，没有技术选型讨论。

### AI 做了什么

AI 首先生成了一份完整的 **开发计划（TODO.md）**，把整个项目拆分成 5 个阶段：

1. **项目初始化** — 技术栈、脚手架、UI 框架
2. **核心学习模块** — 字母、颜色、动物、水果、数字
3. **互动练习** — 听音选图、看图选词、拖拽配对、记忆游戏
4. **游戏化系统** — 关卡地图、奖励、等级、每日目标
5. **打磨与上线** — 音效、动画、引导、兼容性

然后它自己选了技术栈：

- **React 19 + TypeScript** — 现代前端标配
- **Vite 6** — 构建快
- **Zustand** — 轻量状态管理，支持 localStorage 持久化
- **PWA** — 离线可用
- **CSS Modules** — 样式隔离

接着一口气搭好了：
- GitHub 仓库 + SSH Deploy Key
- 多邻国风格的深色主题 UI（卡通配色、大按钮、圆角卡片）
- PWA 配置（manifest + Service Worker）
- iPad 适配（viewport、触控优化）
- ESLint + Prettier 代码规范

### 这阶段的关键决策

**技术选型没让我操心。** AI 根据需求自动判断：儿童 App → 需要流畅动画 → React；iPad 优先 → 需要 PWA 离线；简单状态 → Zustand 比 Redux 合适。

这些都是合理的工程选择。

## 第二阶段：核心学习模块（23:46 - 23:50）

### 4 分钟，5 个模块

这一阶段只用了 4 分钟。AI 写了 5 大学习模块：

1. **🔤 字母学习** — A-Z 闪卡，大写小写，配合语音朗读（Web Speech API）
2. **🎨 颜色主题** — 看图选词，红橙黄绿蓝紫粉黑
3. **🐾 动物主题** — 听音选图，cat / dog / bird / fish…
4. **🍎 水果主题** — apple / banana / grape / orange…
5. **🔢 数字主题** — 1-10 数数，看图识数

每个模块都有统一的学习流程：
```
展示 → 跟读 → 练习 → 反馈
```

### 语音能力

最让我惊喜的是，AI 直接用了 **Web Speech API** 实现发音，不需要任何外部语音文件或 API Key。浏览器原生的 TTS 能力就能搞定。

## 第三阶段：互动练习（23:50 - 23:54）

### 4 种练习模式，也是 4 分钟

学完要练，AI 给了 4 种互动模式：

1. **🔊 听音选图** — 听英文发音，从 4 张图中选对的
2. **👁️ 看图选词** — 看图片，从 4 个单词中选对的
3. **✋ 拖拽配对** — 把单词拖到对应图片上
4. **🧠 记忆游戏** — 翻牌配对，经典连连看

每个练习都有：
- 答对/答错的即时反馈动画
- 进度条和得分
- 练习结束的结果页（星星评价）

### 拖拽实现

拖拽配对是最复杂的，AI 用了 HTML5 Drag & Drop API，同时做了触控兼容（iPad 上的 touch 事件）。这在纯前端里不算简单，但它处理得很好。

## 反思：人机协作的正确姿势

### 我做了什么？

说实话，我做的事很少：

1. **提了一个需求**（一句话）
2. **确认了开发计划**（看了一眼 TODO，说 OK）
3. **说了 5 次"继续"**

整个过程中，我没有：
- ❌ 写一行代码
- ❌ 调过一个 bug
- ❌ 配过一个环境
- ❌ 画一张原型图

### AI 做了什么？

AI 做了几乎所有"体力活"：
- 需求拆解和技术选型
- 编写全部 11000+ 行代码
- 处理兼容性问题
- 游戏化系统设计
- Git 版本管理

### 这种模式适合什么项目？

**适合的：**
- ✅ 你清楚自己想要什么（"做个类似 X 的 Y"）
- ✅ 项目有成熟的参考范式（多邻国 → 游戏化学习 App）
- ✅ 技术栈主流（React / Vue / Next.js 等生态成熟）
- ✅ 主要是前端展示类项目

**不适合的：**
- ❌ 需求模糊、需要大量探索的产品
- ❌ 涉及复杂后端架构和数据库设计
- ❌ 需要深度领域知识（金融、医疗等）
- ❌ 依赖特定硬件或外部系统

### 最大的启示

**AI 不会替代程序员，但会 dramatically 改变程序员的工作方式。**

以前我做一个这样的项目，可能需要：
- 1 天写需求文档
- 3-5 天开发
- 2 天调 bug 和优化

现在？**1 个小时。**

但这不代表程序员没用了。恰恰相反：

- **需求定义能力**变得更重要了——你得能说清楚要什么
- **架构判断力**变得更重要了——AI 做出的方案你得能评估
- **品味和审美**变得更重要了——AI 写的代码你需要把控质量

本质上，程序员从"手艺人"变成了"导演"。
